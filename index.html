The frame

60 / 30 / 10

Most problems are not AI problems. Roughly 60% is traditional code and database work. 30% is rule-based logic. 10% is genuine AI. Teams who skip that distinction spend budget putting language models where Postgres would do the job faster, cheaper, and with better guarantees.

  • 60%Traditional code and database work
  • 30%Rule-based logic
  • 10%Genuine AI

A crisis management playbook has parts that belong on each layer. A customer journey has parts that belong on each layer. The value is knowing which is which before anyone writes code.

Applied to your practice specifically: the workshop IP you have built, the journey maps you have drafted, the crisis templates you are already shipping, all of those become the rule-based and context layers of an orchestration stack. The agent layer sits on top. You do not have to become an engineer to sell this. You have to become the person who decides which layer the client's problem belongs on.

That is the methodology. We published it. It is called Interpretable Context Methodology, in ACM TiiS. The repo is open. The short version: agent context can be organized as a folder structure a non-engineer can read and edit, which means it can be taught to your clients, not just built for them. Repo: github.com/RinDig/Interpretable-Context-Methodology-ICM-.
